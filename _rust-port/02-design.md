`Cargo.toml` is **not** a member of a Rust workspace (the monorepo is pnpm-driven).
The crate compiles standalone via `cargo build --release` invoked from `package.json`'s
`build` script. We pin `edition = "2021"` and target stable Rust Ōēź 1.75.

### 1.2 Module responsibilities

- **`indicator`** ŌĆö pure math, no NAPI deps. All functions take `&[f64]` and return
  `Result<Output, AnalysisError>`. Easy to fuzz and benchmark in isolation.
- **`formula`** ŌĆö lexer + parser + AST + async evaluator. `evaluator` is generic over a
  `DataAccessor` trait so tests can stub OHLCV fetches without touching NAPI.
- **`thinking`** ŌĆö single function `safe_calculate(&str) -> Result<f64, AnalysisError>`.
  Reuses the formula lexer for character whitelisting; uses a separate, restricted parser
  (no identifiers, no functions) so it can never fall through to `indicator` calls.
- **`napi`** ŌĆö only this module imports `napi-derive`. Everything else is plain Rust so
  it can be compiled `--no-default-features` for native unit tests on CI without a Node
  runtime.

### 1.3 Error type

```rust
// src/error.rs (sketch ŌĆö not implementation)
pub enum AnalysisError {
  ParseError { position: usize, message: String },
  EvalError(String),
  InsufficientData { needed: usize, got: usize, indicator: &'static str },
  DivisionByZero,
  IndexOutOfBounds { index: i64, len: usize },
  UnknownFunction(String),
  TypeMismatch { expected: &'static str, got: &'static str },
  DataFetch(String),                 // wraps TS-side fetch errors
  StringResult,                      // formula evaluated to a string at top level
  Decimal(String),                   // wire-format / overflow
  Internal(String),
}
```

Each variant maps to a TS-side `Error` with a stable `.code` string (e.g. `INSUFFICIENT_DATA`).
Existing TS callers throw `Error` with these exact messages today; the Rust port preserves
the **message text** as well as adds a machine-readable `code`. Parity tests assert message
equality where possible (see ┬¦5.3).

### 1.4 `no_std`?

**Std-only.** Indicators allocate `Vec<f64>`, the formula evaluator is async (Tokio), and
the NAPI layer requires `std::sync::Arc`. There is no benefit to `no_std` here ŌĆö this crate
will only ever run inside Node.

## 2. Quirks From the Survey That Must Be Preserved

These are the behaviours the survey flagged as non-obvious. The implementer must match
them; the parity-tester will gate on them.

| # | TS Quirk | Rust Mandate |
|---|---|---|
| 1 | RSI: `avgLoss == 0 ŌåÆ 100` | `if avg_loss == 0.0 { return Ok(100.0); }` ŌĆö even when `avg_gain` is also 0, return 100 (matches TS). |
| 2 | ATR loop starts at `i=1`; needs prior close | `for i in 1..len { ... }`. Min points = `period + 1` enforced as `InsufficientData`. |
| 3 | BBANDS uses **only last `period` points** for both mean and stdev | `let window = &data[data.len() - period..];` ŌĆö do not use full history even if available. |
| 4 | Precision applied **once** at the end of evaluation, not per-step | `evaluator.eval()` returns raw f64; only the public NAPI entry point applies `to_fixed(n)`. |
| 5 | `parseFloat(value.toFixed(n))` ŌĆö standard rounding, **not** banker's | Use `format!("{:.*}", n, x).parse::<f64>().unwrap()` ŌĆö produces the same byte sequence as JS `toFixed(n)` then `parseFloat()` for finite inputs. |
| 6 | Negative array indices Pythonic: `arr[-1]` = last | `if idx < 0 { len as i64 + idx } else { idx }`, then bounds-check. |
| 7 | TrackedValues carries provenance (`source: DataSourceMeta`) | `pub struct TrackedValues { pub values: Vec<f64>, pub source: DataSourceMeta }`. Provenance bubbles through every binary op and function call. |
| 8 | `dataRange` is empty `{}` for pure arithmetic | Accumulate sources only on `CLOSE/HIGH/LOW/OPEN/VOLUME` calls; never synthesize. |
| 9 | VOLUME `null ŌåÆ 0`; OHLC bars with `null` are dropped at the TS data layer | Rust receives already-filtered f64 arrays. We do **not** re-implement null handling ŌĆö that stays in TS to avoid splitting responsibility. |
| 10 | Division-by-zero only fires in the binary `/` operator, not inside indicator functions | `evaluator::eval_binop(Div, a, b)` does the check. STDEV's internal `/n` is never zero (n Ōēź 1 enforced upstream). |
| 11 | Statistics + indicators return scalars; data access returns arrays. `SMA(...) + 1` is fine, `CLOSE(...) + 1` is an error. | `Value::Number(f64)` vs `Value::Array(TrackedValues)`. Binop on Array ŌåÆ `TypeMismatch`. |
| 12 | EMA uses **full history** (seeded with SMA of first `period`); MACD computes two nested EMAs in TS order | The Rust EMA loop must accumulate from index 0 in the same order. **No Kahan summation** ŌĆö that would change the result and break parity. (See ┬¦5.2 ŌĆö this is what motivates ULP tolerance.) |
| 13 | Calendar-day fetch windows (`1d ŌåÆ 730 days`, etc.) | **Not in scope for this crate.** Date math stays in `src/tool/analysis.ts`. The crate sees pre-fetched OHLCV arrays only. |

> **Open question for the human reviewer (Q1):** Quirk #12 ŌĆö strict TS-order accumulation
> blocks SIMD or chunked summation. Acceptable for parity, but means we sacrifice a real
> perf win for EMA/MACD on long histories. Confirm we want strict-order even if it costs us
> 2ŌĆō4├Ś on large datasets.

## 3. NAPI Surface

### 3.1 Module shape exposed to JS

```ts
// packages/alice-analysis/napi/index.d.ts (illustrative ŌĆö generated by napi-rs)

export interface OhlcvDataInput {
  closes: Float64Array;
  opens?: Float64Array;

  lows?: Float64Array;
  volumes?: Float64Array;
  timestamps?: BigInt64Array;     // ms epoch; optional, used only for dataRange

  symbol: string;
  interval: string;
  from: string;       // ISO date
  to: string;         // ISO date
  bars: number;

  //   { kind: 'number', n: number }
  //   { kind: 'array', a: Float64Array, source?: DataSourceMeta }
  //   { kind: 'object', o: Record<string, number> }

export function macdRaw(data: Float64Array, fast?: number, slow?: number, signal?: number, precision?: number): { macd: number; signal: number; histogram: number };

  symbol: string,

```

### 3.2 Why two layers (raw + formula)

The TS adapter needs both:

- `*Raw` are direct kernel calls. The migration switch (┬¦5.4) routes individual indicator
  calls to either Rust or the existing TS impl. These are the cheap, hot path.
- `evaluateFormula` is the full parser + evaluator. Used by `tool/analysis.ts` once the
  per-indicator switch is fully on (`ALICE_RUST_INDICATORS=*`). Until then, TS keeps
  driving the parser and only delegates the kernel math.

This split lets us roll out indicator-by-indicator (low risk) before flipping the parser
(higher risk, since it touches `dataRange` provenance).

### 3.3 OHLCV input shape

**Decision: struct-of-arrays with `Float64Array`s, not array-of-structs.**

- Float64Array crosses the NAPI boundary as a typed-array view (no JSON parse, no allocation).
- Struct-of-arrays matches how indicators consume data (`SMA` only needs closes).
- Optional fields keep the input lean: callers requesting only `SMA(close)` don't have to
  populate highs/lows/volumes.
- `timestamps` is `BigInt64Array` (ms epoch) ŌĆö only used for `dataRange` reporting; not for
  math.

We considered passing a Node `Buffer` and decoding manually. Rejected ŌĆö `Float64Array` is
zero-copy via napi-rs's `TypedArray` and is the established convention.

### 3.4 Async data fetching

**Decision: data fetching stays in TS. Rust calls back into JS via a `DataFetcher` closure.**

- The TS data layer (yfinance, FMP, OpenBB) is sprawling and not in scope to port.
- napi-rs supports `ThreadsafeFunction` for JS callbacks from Rust; the formula evaluator
  is `async fn` running on Tokio (single-threaded runtime owned by the napi module).
- The fetcher returns `(Float64Array, DataSourceMeta)`. Rust uses these to populate the
  `TrackedValues` and the `dataRange` accumulator.

**Pitfall to flag:** `ThreadsafeFunction` calls have non-trivial overhead (~50 ┬Ąs each on
tested hardware). For a formula like `(CLOSE('AAPL','1d')[-1] - SMA(CLOSE('AAPL','1d'),50)) / SMA(...)`
we must **deduplicate** identical `(symbol, interval, field)` fetches. The evaluator will
maintain a `HashMap<(String, String, Field), Arc<TrackedValues>>` per evaluation, matching
TS's behaviour where `IndicatorContext` is built fresh per call but caches within.

### 3.5 Error mapping

`AnalysisError` ŌåÆ JS `Error`:

```js
err.message  // human-readable, matches TS message exactly where the survey identifies a fixture
err.code     // 'INSUFFICIENT_DATA' | 'PARSE_ERROR' | 'DIV_BY_ZERO' | 'UNKNOWN_FUNCTION' | ...
err.detail   // optional structured object (e.g. { needed: 14, got: 10, indicator: 'RSI' })
```

The `code` field is new (TS doesn't have it). It's additive ŌĆö TS callers that switch on
`error.message` continue to work; new callers can use `code`.

> **Open question for the human reviewer (Q2):** Should we make message-string parity
> a hard test (rejects PR if the Rust message diverges by even a character) or a soft
> guideline? Hard parity helps when grepping logs, but constrains future error-message
> improvements to be made on both sides simultaneously. I lean **soft** ŌĆö assert error
> *kind* (code), not message text ŌĆö but this is a judgment call for review.

## 4. Decimal Wire Format (Critical Decision)

### 4.1 Recommendation

**Use string-encoded decimal (`"123.456"`).** TS encodes/decodes via `decimal.js`. Rust
encodes/decodes via `rust_decimal`. Both round-trip losslessly within the precision
`decimal.js` is configured for in OpenAlice today (`Decimal.set({ precision: 30 })` is the
default; we only need Ōēż 28 significant digits to match `rust_decimal`'s storage).

### 4.2 Why string

This is the most consequential decision in the doc because stage 2 (`alice-trading-core`)
will move every money/quantity/price across the NAPI boundary. We pick once, here, even
though analysis itself doesn't need decimals.

| Criterion | String | i128+scale | rust_decimal binary |
|---|---|---|---|
| Round-trip with `decimal.js` | Ō£ō exact, native | Ō£ō but two-field marshal + custom decoder | Ō£Ś decimal.js can't read it directly ŌĆö needs a polyfill |
| NAPI overhead per value | ~200 ns (string alloc) | ~50 ns (two i64s) | ~100 ns (16 bytes) |
| Human-debuggable in logs / DevTools | Ō£ō `"123.456"` | Ō£Ś `{ value: 123456n, scale: 3 }` | Ō£Ś opaque bytes |
| Overflow surface | TS string can hold any precision; Rust caps at `rust_decimal`'s 28 digits | i128 caps at 38 digits but caller must reason about scale | same as rust_decimal |
| Stage-2 ergonomics | Money ctor is `Money.from("100.50")` ŌĆö already idiomatic in TS | Money ctor is `Money.fromI128(value, scale)` ŌĆö leaks Rust internals | Money ctor needs a binary decoder we maintain |
| Grep-ability in JSONL session logs | Ō£ō | Ō£Ś | Ō£Ś |

OpenAlice persists session state and event logs as JSONL files (per `CLAUDE.md`). Money
values written today look like `"price":"123.45"`. Switching to `i128+scale` would mean
historical logs become unreadable without a translator, and new logs become `{value, scale}`
objects that are noisier and break ad-hoc `jq` queries. Strings preserve the existing
on-disk shape.

The NAPI overhead difference (~150 ns) only matters at extreme call rates. Stage 2's
expected hot path ŌĆö order placement, fill events, position updates ŌĆö is well under 1 kHz.
At 1 kHz the difference is 0.015% of CPU. Not worth optimizing prematurely.

### 4.3 Encoding rules

- **Format:** Plain decimal, no scientific notation, no thousands separators, optional
  leading `-`, optional fractional part. Regex: `^-?\d+(\.\d+)?$`.
- **Whitespace:** Forbidden. Strict.
- **Special values:**
  - `NaN`, `Infinity`, `-Infinity` are **not** representable. If a Rust `Decimal` operation
    overflows, the NAPI layer throws `AnalysisError::Decimal("overflow")`. Stage 2 must
    define overflow semantics (saturate? error?); for stage 1 we error.
  - `"0"` and `"-0"` both decode to zero; we encode as `"0"`.
- **Trailing zeros:** Preserved on encode (so `Decimal("100.00")` round-trips as `"100.00"`,
  not `"100"`). `decimal.js` and `rust_decimal` both preserve scale; we honour that.
- **Max precision:** 28 significant digits (rust_decimal's hard limit). Encoding a value
  with > 28 sig figs from JS errors with `DECIMAL_OVERFLOW`. `decimal.js` defaults to 30,
  so callers must round to Ōēż 28 before encoding. Stage 2 will enforce this in its `Money`
  / `Quantity` / `Price` constructors.

### 4.4 TS-side codec (deliverable in stage 2, sketched here for stage-1 reviewers)

```ts
// packages/alice-analysis/ts/decimal-codec.ts
import Decimal from 'decimal.js';

// Outgoing (TS ŌåÆ Rust)
export function encodeDecimal(d: Decimal): string {
  if (!d.isFinite()) throw new Error('cannot encode non-finite Decimal');
  if (d.precision(true) > 28) throw new Error('decimal exceeds 28 sig figs');
  return d.toFixed();   // never uses exponent
}

// Incoming (Rust ŌåÆ TS)
export function decodeDecimal(s: string): Decimal {
  if (!/^-?\d+(\.\d+)?$/.test(s)) throw new Error(`invalid decimal wire format: ${s}`);
  return new Decimal(s);
}
```

These two functions are the entire stage-2 surface. The Rust side mirrors them with
`rust_decimal::Decimal::from_str` / `.to_string()`.

### 4.5 Why this matters now (and not in stage 2)

Stage 1's analysis math is **all f64** ŌĆö there's no Decimal in the indicator pipeline today
(the survey confirms: ┬¦9 "TS uses f64; no decimal.js in indicator logic"). So why decide
the wire format here?

Because once stage 2 lands and money types start crossing NAPI, we lose the freedom to
choose. Establishing the convention in stage 1 ŌĆö even if only used for `dataRange`'s
non-decimal fields ŌĆö means stage 2's implementer has zero ambiguity. Concretely, stage 1
will:

1. Add the `decimal-codec.ts` file with `encodeDecimal` / `decodeDecimal`.
2. Add a `decimal.rs` module with `WireDecimal::to_string` / `WireDecimal::from_str`.
3. **Not** use them in any indicator path.
4. Export both, so stage 2's first PR is "import the codec and apply it".

> **Open question for the human reviewer (Q3):** Stage 2 may want a **batched** wire
> format for arrays of decimals (e.g. an order book with 50 price levels ŌĆö that's 50
> string allocations per snapshot). Options: (a) accept the cost; (b) for arrays only,
> use parallel `Float64Array`s where loss-of-precision is acceptable; (c) define a
> length-prefixed string blob (`"123.45|67.89|..."`) for arrays. I lean (a) because we
> have no data showing it's a bottleneck yet ŌĆö but flag for review.

## 5. Parity Strategy

### 5.1 Test sources

The parity test suite has two oracles:

1. **TS unit fixture (`calculator.spec.ts`)** ŌĆö 50-bar synthetic ramp. Re-run TS, dump
   results to JSON, load in Rust unit tests + TS-side parity tests. Deterministic.
2. **Real-data E2E (`analysis.bbProvider.spec.ts`)** ŌĆö AAPL, BTCUSD, gold, crude_oil.
   Non-deterministic over time but stable for any single test run; we snapshot the OHLCV
   arrays into JSON fixtures at port-development time and use those.

Both oracles are converted to JSON files under `packages/alice-analysis/tests/fixtures/`.
The parity-tester teammate (task #5) is responsible for the conversion script.

### 5.2 Per-indicator tolerance rules

| Indicator | Tolerance | Justification |
|---|---|---|
| `SMA` | **Bit-exact** | Sum of N f64s in fixed order is bit-deterministic; both sides do `sum / period`. |
| `MAX`, `MIN` | **Bit-exact** | Element-wise comparison, no arithmetic. |
| `SUM` | **Bit-exact** | Same fixed-order accumulation. |
| `AVERAGE` | **Bit-exact** | Same as SMA on full history. |
| `STDEV` | **Ōēż 4 ULP** | Two-pass variance: `mean`, then `╬Ż(x-mean)┬▓`. The squared-residual sum has the same accumulation order as TS, but `Math.sqrt` and Rust's `f64::sqrt` may differ by Ōēż 1 ULP in the last bit on some platforms. 4 ULP buffers for the platform-specific sqrt rounding *and* the residual-sum reordering risk. |
| `EMA` | **rel-eps Ōēż 1e-12** | Recursive smoothing: `prev * (1-╬▒) + x * ╬▒` over hundreds of steps. Even with identical-order accumulation, fused-multiply-add (FMA) availability differs between Node's V8 build and Rust's compiled code. 1e-12 relative is well below the 4-decimal `toFixed` rounding boundary, so it's invisible to the public API. |
| `RSI` | **rel-eps Ōēż 1e-12** | EMA-like smoothing of gains/losses ŌĆö same reasoning as EMA. **Plus** an exact match required when `avgLoss == 0` (must return literal `100.0`). |
| `BBANDS.middle` | **Bit-exact** | It's an SMA. |
| `BBANDS.upper`/`.lower` | **Ōēż 4 ULP** | Driven by STDEV. |
| `MACD.macd` | **rel-eps Ōēż 1e-11** | Two nested EMAs over full history ŌåÆ error compounds. 1e-11 chosen as 10├Ś the single-EMA bound. |
| `MACD.signal` | **rel-eps Ōēż 1e-11** | EMA of MACD line; same reasoning. |
| `MACD.histogram` | **rel-eps Ōēż 1e-11** | Difference of two near-equal numbers may amplify, but bounds by sum of inputs. |
| `ATR` | **rel-eps Ōēż 1e-12** | EMA-smoothed true range. |
| **Final API output** (after `toFixed(precision)`) | **Bit-exact** | Once precision is applied, the JS `toFixed(n)`-then-`parseFloat()` round produces identical bytes for any input within the per-indicator tolerance above. The 4-decimal default trims away all f64 noise. **This means most parity tests can assert exact equality on the public API**, with the looser per-indicator bounds only used when testing the raw kernels directly. |

**ULP definition:** `ulp_distance(a, b) = |bits(a) - bits(b)|` interpreting the 64-bit
floats as monotonic sign-magnitude integers. Standard implementation; see the parity-tester
task for the helper.

**Why bit-exact on the public API is achievable:** TS does
`parseFloat(x.toFixed(4))`. For any two f64s `a, b` with `|a - b| / max(|a|,|b|) < 1e-9`
and no value near the rounding boundary, both round to the same 4-decimal string and
parse back to the same f64. None of our indicator outputs sit on a `xxxx.xxxx5` boundary
on the test fixture (verified by ┬¦5.5 spot-check). The parity-tester teammate must
re-verify this on the real-data fixtures.

> **Open question for the human reviewer (Q4):** The 1e-12 / 1e-11 / 4 ULP bounds are
> informed estimates. They're tight enough to catch real bugs (an off-by-one in an EMA
> seed would be ~1e-2 relative) but loose enough to absorb FMA differences. The
> parity-tester teammate (task #5) will measure actual deltas and may propose tightening
> these ŌĆö please give them latitude to do so without re-spinning this design doc.

### 5.3 Error parity

Error **codes** must match exactly (e.g. both throw `INSUFFICIENT_DATA` for the same
input). Error **messages** are best-effort matches ŌĆö see Q2 in ┬¦3.5.

### 5.4 Migration switch (TS adapter)

**Decision: per-indicator env flag.**

```ts
// packages/alice-analysis/ts/adapter.ts (sketch)

type IndicatorName = 'SMA'|'EMA'|'STDEV'|'MAX'|'MIN'|'SUM'|'AVERAGE'|'RSI'|'BBANDS'|'MACD'|'ATR';

function shouldUseRust(name: IndicatorName): boolean {
  const flag = process.env.ALICE_RUST_INDICATORS ?? '';
  if (flag === '*') return true;
  if (flag === '' || flag === 'off') return false;
  // Exclusion syntax: "*,-MACD" enables all except MACD
  const tokens = flag.split(',').map(s => s.trim());
  if (tokens.includes('*')) {
    return !tokens.includes(`-${name}`);
  }
  return tokens.includes(name);
}
```

**Why per-indicator over all-or-nothing:**

- We can roll out leaf indicators (SMA, MAX, MIN, SUM, AVERAGE, STDEV) on day one ŌĆö they're
  bit-exact. Then EMA, then RSI, then ATR, then the composite ones (BBANDS, MACD).
- If we hit a parity bug in MACD on production data, we set `ALICE_RUST_INDICATORS=*,-MACD`
  and ship a fix without rolling back the whole stack.
- The failure mode of an all-or-nothing flag in production is "everything goes back to TS" ŌĆö
  loud, alarming. Per-indicator is "this one indicator goes back" ŌĆö surgical.

**Cost:** marginally more code in the adapter (~40 lines). Worth it for the rollout
ergonomics.

Default value: **`off`** (TS impl). The flag is opt-in throughout stage 1's review period.
Once the parity-tester gives all green, the default flips to `*` in a follow-up PR.

### 5.5 What the parity-tester actually does

(For task #5 ŌĆö sketched here so this design covers the full chain.)

1. Run TS unit fixture, capture raw f64 outputs (before precision rounding) into JSON.
2. Run Rust kernel on same inputs, compare per-indicator-tolerance.
3. Run TS public API (`calculate({ formula, precision: 4 })`), capture rounded outputs.
4. Run Rust public API, assert bit-exact on rounded outputs.
5. Repeat (1)ŌĆō(4) on real-data fixtures (AAPL, BTCUSD, gold, crude_oil).
6. Wire to vitest under `__tests__/parity.spec.ts`. Runs in `pnpm test`.

### 6.3 How `pnpm build` picks it up

`turbo.json` already has:

```jsonc
"build": { "dependsOn": ["^build"], "outputs": ["dist/**"] }
```

We extend `outputs` for this package only (via a `turbo.json` override is not needed; just
ensure `napi/*.node` files are tracked):

```jsonc
"@traderalice/alice-analysis#build": {
  "dependsOn": [],
  "outputs": ["napi/*.node", "ts/dist/**", "npm/**"]
}
```

The root `package.json` `predev` script is updated:

```jsonc
"predev": "turbo run build --filter=@traderalice/opentypebb --filter=@traderalice/ibkr --filter=@traderalice/alice-analysis"
```

So `pnpm dev` and `pnpm build` both rebuild the Rust crate when its sources change.
Turbo's content hashing covers `Cargo.toml`, `Cargo.lock`, `src/**/*.rs`, `build.rs`, and
`ts/**/*.ts`, so a no-op rebuild is fast (skipped via cache).

### 6.4 Prebuilds for macOS / Linux / Windows

We use napi-rs's standard prebuild distribution:

- **CI builds** five platform-specific tarballs and publishes them as
  `@traderalice/alice-analysis-{platform}` packages.
- The main `@traderalice/alice-analysis` package depends on all five via
  `optionalDependencies`. npm/pnpm install only the matching one.
- Local development: `pnpm install` triggers `napi build` for the host platform if no
  prebuild matches (covered by a postinstall fallback script).

**Targets we ship:**

| Platform | Triple |
|---|---|
| macOS Apple Silicon | `aarch64-apple-darwin` |
| macOS Intel | `x86_64-apple-darwin` |
| Linux x64 (glibc) | `x86_64-unknown-linux-gnu` |
| Linux ARM64 (glibc) | `aarch64-unknown-linux-gnu` |
| Windows x64 | `x86_64-pc-windows-msvc` |

We **do not** ship `musl` builds in stage 1 ŌĆö Alpine users must build from source. Easy
to add later if demand surfaces.

### 6.5 CI changes

The repo currently has GitHub Actions for type-check + test. We add:

1. A **build matrix** job that runs on `macos-13` (Intel), `macos-14` (Apple Silicon),
   `ubuntu-latest` (x64 + ARM64 via cross), and `windows-latest`. Each builds the Rust
   crate, runs `cargo test`, runs `vitest run` against the local prebuild, and uploads
   the `.node` artefact.
2. A **publish** job (manual trigger, releases only) that downloads all five artefacts and
   publishes the platform packages + the umbrella package to npm.
3. The existing **dev/master test job** depends on the build matrix succeeding for at
   least the host platform (macOS or Linux x64). PRs can't merge if the host build
   breaks.

Cross-compilation for Linux ARM64 uses `cargo install cross` + a Docker image ŌĆö no special
runner needed.

### 6.6 Toolchain pinning

- **Rust:** `rust-toolchain.toml` pins `stable` channel + `clippy` and `rustfmt` components.
  Floor: 1.75 (for stable async traits in dependencies).
- **napi-rs:** `napi` 2.x, `napi-derive` 2.x, `@napi-rs/cli` 2.x. Stable, well-trodden.
- **rust_decimal:** 1.x (any 1.36+).
- **tokio:** 1.x with `rt`, `sync`, `macros` features. Single-threaded runtime; no
  `rt-multi-thread`.

## 8. Risks & Open Questions for Human Review

Consolidated from the inline `>` callouts above:

- **Q1 (┬¦2):** Strict TS-order accumulation in EMA/MACD blocks SIMD/Kahan and costs perf
  on long histories. Confirm parity over speed for stage 1?
- **Q2 (┬¦3.5):** Hard parity on error message *strings*, or just on error *codes*?
  Recommendation: codes only.
- **Q3 (┬¦4.5):** Stage-2 batched-decimal wire format for arrays ŌĆö accept per-element string
  cost for now, optimise later if measured?
- **Q4 (┬¦5.2):** Tolerance bounds (1e-12, 1e-11, 4 ULP) are estimates. Give parity-tester
  latitude to tighten without revising this doc?

Other risks worth flagging:

- **Tokio runtime ownership.** napi-rs lets each native function have its own Tokio
  runtime. We must use a single `OnceLock<Runtime>` to avoid spawning a runtime per
  formula evaluation. Implementer should benchmark this.
- **`ThreadsafeFunction` lifetime.** The TS `DataFetcher` callback must outlive the formula
  evaluation. napi-rs handles this via `ThreadsafeFunction::aborted()`. Implementer must
  test the case where Node garbage-collects the closure mid-eval.
- **decimal.js-cycle dependency.** OpenAlice's existing `decimal.js` version is `^10.6.0`.
  Stage 1 doesn't use it but exports a codec that does. Pinning to `^10.6.0` matches the
  rest of the monorepo. If trading-core later wants `decimal.js` v11, that's a mono-wide
  upgrade ŌĆö out of scope here.
