# ADR-003: TypeScript <-> Rust Binding Strategy

- Status: Accepted (planning-only; no source code authorized by this ADR)
- Date: 2026-04-28
- Authors: CTO / Program Orchestrator (drafting); Architecture Lead (review owner)
- Supersedes: none
- Related: [ADR-001](./ADR-001-rust-boundary.md), [ADR-002](./ADR-002-feature-flag-policy.md), [Playbook section 10.1](../PAPERCLIP_OPENALICE_RUST_REFACTOR_PLAYBOOK.md), module contracts for [analysis_core](../module-contracts/analysis-core.md), [trading_core](../module-contracts/trading-core.md), [store_core](../module-contracts/store-core.md)

## Context

Rust crates need a stable, low-overhead, JSON-friendly bridge into the existing OpenAlice Node.js process. Phase 1 contracts already require:

- DTOs are JSON-compatible at the binding boundary
- numeric, nullable, and missing values are handled deterministically
- error types from Rust are normalized before reaching the TypeScript tool surface
- the Rust path can be disabled per-module without code changes (see [ADR-002](./ADR-002-feature-flag-policy.md))

What Phase 1 did not pin: the binding mechanism itself, who owns DTO authoring, where the binding package lives, how synchronous vs. asynchronous calls are handled, and what the failure-isolation contract is.

## Decision

### Default binding mechanism

Use **Node-API (N-API) via `napi-rs`** for in-process bindings to all three Rust crates.

Reasons:

- `napi-rs` is the most mature Rust <-> Node-API toolchain in the ecosystem at decision date and is widely used in production by similar shape projects.
- Node-API is ABI-stable across Node major versions; OpenAlice can upgrade Node without rebuilding bindings.
- In-process calls avoid IPC, serialization round-trips through the OS, and a separate process supervisor - all of which would dominate the per-call overhead for small operations.
- The same toolchain handles macOS / Linux / Windows host targets that OpenAlice already supports.

### When in-process is **not** acceptable

`store_core` may, after parity proof, optionally route some workloads (e.g., long-running archive scans) through a **sidecar process** if and only if:

- per-call overhead under in-process bindings exceeds the threshold recorded in the `store_core` benchmark plan, **and**
- a future ADR explicitly approves the sidecar for the named workloads, **and**
- the legacy TypeScript path remains the rollback target (the sidecar is an alternative Rust path, not a replacement for the legacy fallback).

`analysis_core` and `trading_core` use in-process bindings only. A sidecar for either requires a new ADR.

### Package layout

Each Rust module owns exactly two units:

```
crates/<module>/                          # pure Rust kernels, no Node-API types
  Cargo.toml
  src/lib.rs
  tests/                                  # Rust unit + property tests

packages/node-bindings/<module>/          # Node-API binding crate + npm package
  Cargo.toml                              # depends on crates/<module>
  src/lib.rs                              # napi-rs bridge; DTO conversion only
  package.json                            # npm package consumed by src/domain/<module>/
  index.d.ts                              # generated TypeScript types
  __tests__/                              # binding-level integration tests
```

Rules:

- The crate at `crates/<module>/` must compile and test independently of any Node toolchain. It must not reference `napi`, `neon`, or `node` types.
- The binding crate at `packages/node-bindings/<module>/` is the only place allowed to know about Node-API.
- The npm package's published surface is a small DTO-shaped API consumed by `src/domain/<module>/` adapters. It does not export raw napi values.
- Type definitions (`index.d.ts`) are generated by the binding toolchain; they are checked in to support TypeScript consumers without requiring a build of the binding during `pnpm install`.

### DTO contract

DTOs at the binding boundary follow these rules:

1. **JSON-compatible only.** Inputs and outputs serialize/deserialize losslessly through `JSON.stringify` / `JSON.parse`. No `BigInt`, no `Date` objects, no functions, no circular references, no opaque handles crossing the boundary.
2. **Numeric handling is explicit.**
   - Money and quantity in `trading_core` cross the boundary as **strings** (matching the existing public string monetary contract documented in `src/domain/trading/brokers/types.ts`). Decimal-safe representation is enforced inside Rust; conversion to/from string happens at the binding edge.
   - Indicator and formula numerics in `analysis_core` cross the boundary as JSON numbers unless a module-specific ADR documents an exception.
   - `null`, missing, `NaN`, and `+/-Infinity` semantics for each module are governed by that module's contract; the binding must not silently coerce.
3. **Timestamps follow legacy.** `event_log` and `news` keep epoch-millisecond integers. Sessions keep ISO timestamp strings. The binding must not normalize across modules.
4. **Errors are normalized.** Rust errors are converted to typed JS error objects with stable `name`, `code`, and `message` fields before they cross the boundary. Panics in Rust must be caught at the binding edge and converted to an error of code `INTERNAL_RUST_PANIC`; a panic must not crash the Node process.
5. **No mutable shared state.** The binding does not expose Rust-owned references that survive the call. Each call takes ownership of its inputs and produces fresh outputs.
6. **Synchronous by default.** Binding entry points are synchronous unless the workload exceeds the threshold defined in the module's benchmark plan; long-running calls use `napi-rs` async tasks (Tokio worker threads inside Rust) so the Node event loop is never blocked beyond the per-call budget.

### Build, distribution, and CI

- The binding crate compiles to a platform-specific `.node` artifact via `napi-rs`'s standard build pipeline.
- Local development builds the binding from source as part of `pnpm install` (pre/post-install hook on the binding package).
- CI builds the binding on each supported host triple and runs the binding's `__tests__/` suite. Triples must match what OpenAlice already supports for its existing Node toolchain.
- Prebuilt binaries are out of scope for this ADR. If they become required, a follow-up ADR will define the publish workflow.
- `cargo fmt --all --check` and `cargo clippy --workspace -- -D warnings` are mandatory in CI for the workspace once any crate exists. Crate-only and binding-only test commands must be runnable independently for fast local iteration.

### Failure isolation

- A panic in Rust must not crash Node. The binding catches via `napi-rs`'s panic boundary and surfaces an `INTERNAL_RUST_PANIC` error.
- A loaded binding that fails to initialize (e.g., missing native artifact for the host triple) must surface a clear error when the TypeScript adapter attempts to import it. The TypeScript adapter must fall back to the legacy path automatically when the relevant feature flag (see [ADR-002](./ADR-002-feature-flag-policy.md)) is `0`.
- Memory ownership: the binding never returns Rust-owned pointers; all returned values are owned by the JS runtime after the call returns.

### Versioning

- Each Rust crate is versioned independently (`crates/<module>/Cargo.toml`).
- Each Node binding package follows the same SemVer as its crate but the npm package version is the source of truth for TypeScript consumers.
- A binding may not be imported by `src/domain/<module>/` until the binding's `package.json` declares an explicit version compatible with the crate it links.

## Consequences

- The Rust workspace has a single, predictable shape: three crates and three binding packages, all using the same toolchain and conventions.
- TypeScript consumers see only DTOs and typed errors; they never need to know whether a call ran in Rust or TypeScript.
- The legacy TypeScript path remains the rollback target for every module, with the feature flag from [ADR-002](./ADR-002-feature-flag-policy.md) as the only switch operators need.
- Build-time cost increases on machines without a Rust toolchain. The Adapter & Tooling Engineer issue must install and pin `rustc`/`cargo`, wire `cargo fmt`/`cargo clippy` into CI, and confirm `provision_command` from the manifest succeeds before any binding lands.
- Cross-platform binary distribution is a future problem; for now, every host that runs OpenAlice with a flag set to `1` must have the Rust toolchain available (or pull in a binding artifact built for its triple).

## Rejected alternatives

- **`neon`.** Mature but smaller maintenance footprint than `napi-rs` at decision date; less first-class TypeScript type generation; weaker async ergonomics.
- **WebAssembly (`wasm-bindgen`).** Adds runtime overhead, weakens numeric/IO ergonomics for trading and storage workloads, and offers no Node-side benefits OpenAlice cares about.
- **Pure FFI via `node-ffi-napi` or similar.** Rejected. Lower-level, slower per-call, and lacks the ergonomic DTO bridge `napi-rs` provides.
- **Sidecar process for every module.** Rejected as a default. Per-call IPC overhead would dominate small operations in `analysis_core` and would require a process supervisor that OpenAlice does not currently run.
- **Embed Rust as Node `addon` via `node-gyp` directly.** Rejected. Reinvents what `napi-rs` already gives us and loses Node-API ABI stability guarantees.
- **Expose Rust types directly into TypeScript without DTO normalization.** Rejected. Couples TypeScript callers to internal Rust shape and breaks the public-surface stability rule from [ADR-001](./ADR-001-rust-boundary.md).

This ADR is **planning-only**. It authorizes no source code edits. The first crate, binding, or `Cargo.toml` may only land under a separate, scoped Adapter & Tooling Engineer issue that satisfies its own allowed-file list, integration approval, and the rollout/rollback contract in [ADR-002](./ADR-002-feature-flag-policy.md).
